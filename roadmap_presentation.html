Current State & Path Forward

Two examples show the kind of risk that comes with running on decade-old, unmaintained technology. The platform has run without a major incident so far, but with no maintenance or updates in years, that track record reflects luck more than resilience — it can keep working right up until the moment it doesn't, and neither example below is a fire burning today. Both are the sort of thing that lands without warning on a platform no one is keeping current:

Example — latent fault

A billing bug documented in the platform — a card/billing-address mismatch that can block clients from submitting — has sat unaddressed for years. It isn't currently causing problems, but on a system this old, with no one maintaining it, that's fortune rather than safety. It's exactly what hardening is meant to catch before it resurfaces.

Example — vendor deadlines don't wait

Microsoft retires services and API versions on its own schedule, not ours — TLS 1.0/1.1 support in Storage Accounts, .NET 9 support in App Service, and older Key Vault API versions are all being retired within the next 12 months. We don't believe any of these specifically affect us today, but this is the pattern: vendor-forced deadlines land on aging infrastructure continuously, on someone else's schedule, ready or not.

Decision

Three committed months, then it depends how capital arrives

Only the first ~3 months are committed: Phase 1 hardening and a parallel fundraising push, both wrapping up around November. That next raise is targeted at $3–5M total, and could land all at once or in stages:

  • ~$1M lands first — as its own raise, or as an opening tranche of the $3–5M target. Phase 2's prototype and beta validation run before the full build.
  • The full $3–5M arrives directly — Phase 3's full build starts immediately. Feedback comes from the existing working mock as the build progresses, rather than running a separate, funded Phase 2 client pilot in parallel.
  • Neither has landed yet — we keep operating on the hardened platform, no new build, and keep raising rather than spin up a team we can't sustain.

Nothing past November is a commitment; it's a branch that resolves on how and when the money arrives.

Risk — the $1M tranche lands, the rest doesn't

"Phase 2, then Phase 3" assumes the remaining $2–4M follows before the $1M tranche's buffer runs out. If it stalls instead, this isn't the same as "keep operating as-is" — we'd be supporting a validated prototype for 5–10 beta clients while the rest of the client base — 50–100+ by the time a Legacy Fund is live — stays on the merely-hardened legacy platform, with no funded path to the full build.

What we'd actually do in that scenario isn't decided yet — options include:

  • Pause onboarding
  • Attempt a smaller follow-on raise
  • Fold Phase 2's learnings back into the existing platform instead of a separate full build

This needs an answer before we commit to this path.

Recommendation

Fund Phase 1 now. Which phase comes next depends on how capital arrives.

Why this sequencing

  • Sized to what we actually have. Phase 1 is the only one of the three sized to what this raise leaves for technology. It's aimed at the biggest operational risk — the current platform failing under its own weight — without betting on money we don't have yet. Moving on to Phase 2 or 3 would depend on additional fundraising, not this raise.
  • The next raise runs in parallel, not after. That second raise — targeted at $3–5M total — would be pursued starting now rather than after Phase 1 wraps up. How it lands, not just whether it does, determines what happens next: it could arrive all at once, or in stages. If it arrives in stages, an initial ~$1M tranche of that same target funds Phase 2's prototype and beta validation first, with the full build following once the rest of that capital comes in — Phase 2 isn't necessarily capital on top of the $3–5M, it can be the first slice of it. If the full $3–5M arrives directly instead, the full build starts immediately — using the existing working mock to gather feedback as it's built, rather than running a separate, funded Phase 2 client pilot in parallel.
  • Neither Phase 2 nor Phase 3 is just a technology budget. Scaling past Phase 1 — onboarding 50–100+ clients and standing up Legacy Funds — also means scaling the team: this can't stay a one-person product/technology/operations operation, and there are real, currently unpriced costs in hiring, compute/storage, cybersecurity, and compliance that any Phase 2 or 3 raise needs to account for.
  • Timing is tight. A Legacy Fund could need the platform as early as Q1–Q2 2027 — potentially only one to four months after Phase 1 would finish. That's a thin margin, which is the argument for starting the next raise sooner rather than waiting for more clarity. All of these dates and costs are estimates.
