Working Mock — Build Methodology & Handoff Strategy

How the Lovable working clickable mock was built, why the process was structured the way it was, and what that structure gives the development team at handoff.

July 2026  ·  Owner: Adam McKinstrie  ·  Prepared for engineering handoff

About this project AngelSpan is an investor-relations SaaS platform for startups. Internally, a Performance Analyst (PA) reviews founder updates, enters SMART IR scores (a Bell Mason Diagnostic-based scoring system across Product, Market, Team, and Operations), and distributes polished reports to investors and advisors. Legacy Funds is a companion venture fund product where General Partners (GPs) manage startup portfolios and Limited Partners (LPs) invest — it uses AngelSpan's SMART IR data for fund-level decisions. Both products are covered by this working mock.

The Approach

Build Methodology

The prototype was built using a structured, human-in-the-loop process across three tools: Claude (specification and decision-making), Lovable (AI-powered prototype builder), and GitHub (version control and documentation). The process ran in repeating cycles — one cycle per prompt — from the first feature through Cluster 12.

1
Session setup
Each working session began by reading SESSION.md to re-establish current state — what had been built, which decisions were in effect, and what was next. This was the memory continuity mechanism across sessions, compensating for the fact that no AI tool retains context between sessions by default.
2
Prompt drafting
Claude drafted each Lovable prompt from the master spec for the epic being built, the relevant decision records, the design system, and shared agent context files. Claude reviewed each draft from five perspectives simultaneously — product manager, UX designer, frontend lead, backend lead, and technical architect — before finalizing. This multi-role review was designed to catch conflicts and specification gaps before anything was sent to the build tool.
Parallels recognized practice: cross-functional review at the point where decisions are still being shaped, not after they've been embedded in code
3
Human review and approval
Adam reviewed each drafted prompt before it was sent — reading for correctness, challenging assumptions, and approving or requesting changes. No prompt was sent to Lovable without explicit human review and approval. This is the primary accountability checkpoint in the process: the moment where a human confirmed that the specification being sent matched intended product behavior.
Parallels recognized practice: human accountability is more important, not less, in AI-assisted delivery — AI compresses the feedback loop that traditional development allows
4
Delivery to Lovable
Approved prompts were submitted to Lovable.ai. Lovable interpreted the prompt and made code changes to the working mock, automatically committing them to the main branch on GitHub.
5
Verification
After Lovable completed the build, Adam walked through the working mock at angelspan-bridge-flow.lovable.app to verify that the implementation matched what the prompt specified. Screens, flows, data behavior — checked against the intent of the specification.
6
Correction or advance
If verification revealed a misinterpretation, partial execution, or bug, a correction prompt was written and the cycle repeated from step 2. If verification passed, the session advanced to the next prompt. Correction prompts are not signs of failure — they are the record of where specification needed more precision, and where the build tool made assumptions that had to be overridden. E-08 (Client Lifecycle), for example, required five correction passes; those passes document where the hardest product questions were and how they were resolved.
Parallels recognized practice: restoring the clarification feedback loop that AI-only delivery collapses
7
Decision capture
Whenever a non-trivial choice was made — during drafting, during human review, or after verification — a decision record was written and added to decisions/. This ensured that rationale was captured at the moment it was made, not reconstructed later. 38 decisions were logged across the full build.
Parallels recognized practice: decisions made implicitly become untraceable — capturing them at the moment they're made is the only reliable method
The Accountability Layer

Why the Decision Log Exists

A well-recognized risk in AI-assisted development is that AI does not simply accelerate code — it accelerates decisions. Requirements get interpreted quickly, designs are generated fast, and code follows at speed. Each step involves choices that shape how the system behaves. When those choices happen without an explicit capture mechanism, they become embedded in the codebase without anyone having examined them. The system works as built; the problem is that what it was built on may include assumptions that were never surfaced.

The decision log is the direct response to that risk. Every record in decisions/ has four sections:

  • Context What problem or question prompted the decision — why it needed to be made at all.
  • Options considered What alternatives were evaluated, described plainly enough to understand the trade-offs.
  • Choice What was selected and, where relevant, how it works in the prototype.
  • Consequences What the choice enables, what it constrains, what remains open.
The critical distinction Decision records capture intent, not implementation. The working mock's src/ folder captures implementation. When the two diverge — when Lovable built something that does not perfectly match the decision record — the decision record is what is correct. This is also the correct precedence for the development team: master specs and decision records define what the system is supposed to do; Lovable's code shows one way it was approximated.

For the development team, the decision log is the fastest path to understanding why the system works the way it does — not just what it does. When you encounter a reference like "see D-003" in a specification, that reference explains the reasoning behind a product behavior, not just its mechanics. Reading D-003 before designing the production authentication system will save more time than reading the prototype's auth code.

For the Development Team

Durable vs. Disposable

The most important frame for the development team to hold: the working clickable mock was built to prove user experience and product decisions — not architecture. Every screen and flow at angelspan-bridge-flow.lovable.app represents a validated product decision. The underlying technical implementation was chosen for build velocity and is fully disposable.

Treat src/ as a high-fidelity wireframe, not as a starting point for production code.

What this process gives the development team that a typical handoff does not In a typical AI-assisted build, the output is a working demo and a vague handover conversation. The decisions that shaped the product are distributed across chat threads, memory, and the original developer's head — none of it traceable. This build was structured so the inverse is true: the decisions are explicit, numbered, and cross-referenced. A developer picking up any D-ID gets the full rationale for the product behavior behind it — the question that prompted it, the options considered, the choice made, and what it constrains going forward. They do not have to reconstruct reasoning from code behavior. This is the primary value this methodology creates for the handoff.

Visibility Layer

Dashboard Evolution

The build dashboard exists to make the project's status, decisions, and open items visible to stakeholders who are not in every working session. Its appropriate form changes as the team grows. The stages below reflect a pragmatic progression — each stage is triggered by a real change in team size or infrastructure capability, not by a fixed timeline.

Now
Stage 1
Static HTML
Now → engineering handoff
Data baked into HTML. One PM, one repo, GitHub Pages. Zero infrastructure cost. Manual update cadence. Decisions visible; attribution via git history.
Skip
Stage 2
JSON Shell
Not recommended
Data in a separate JSON file; HTML renders it. Adds complexity without eliminating manual updates. Useful life too short to justify. Go directly to Stage 3.
Stage 3
CMS-Backed
When team > 3 contributors
Epics, features, decisions live in Notion or Confluence. Dashboard fetches live data. Non-technical contributors can update. Attribution and sign-off become trackable fields.
Stage 4
Platform Native
Post-platform launch
Dashboard module inside the built platform, pulling from the same database that powers PA workflows and SMART IR scoring. Full integration, no separate tool.

The working mock answered the hardest product questions — does the information architecture work across ten roles spanning two brand contexts, does the dual-brand strategy hold, does the SMART IR scoring UX communicate what it needs to communicate, is the Legacy Fund application flow coherent — at relatively low cost. It did not answer production engineering questions; those start fresh with the development team. The structured build process — human review at every step, decisions captured at the moment they were made, explicit separation of intent from implementation — means the development team inherits a clear picture of why the product works the way it does, not just that it does.
