The Full Build — Implementation Plan

A distilled version of the March 2026 planning document the current three-phase roadmap draws from — what a fully funded $3–5M rebuild would actually cover, once that raise lands.

Source: AngelSpan Unified Platform Product Development Plan, v4 (March 2026) ↗ · Distilled July 2026 · Access limited to Joe, Matt, and Alec
ℹ️

How this relates to the rest of the roadmap. This document predates the current $500k bridge raise and the Phase 1 → Phase 2 → Phase 3 staging used elsewhere on this site. It describes, in more engineering-level detail, what the eventual $3–5M rebuild (summarized as "Phase 3" on the roadmap) would include. Its own internal stages are called Milestones below, not Phases, specifically so the two numbering schemes don't collide.

Background

What Exists Today

The roadmap already covers the business case for hardening; the source document adds sharper technical specifics worth carrying forward:

  • Frontend is Angular 1, end-of-life since January 2022 — no further security updates are possible.
  • Backend is .NET Core 2.2 plus separate Java services — .NET Core 2.2 itself has been unsupported since December 2019, over six years unpatched.
  • Two live Stripe bugs: subscriptions can activate without a confirmed payment, and credit card updates can create orphaned payment records.
  • No MFA, no multi-user client accounts, no self-service password reset — every one of these requires manual engineering intervention today.
  • No SOC 2, no documented backup/disaster-recovery plan, no audit logging — all three are gaps for any enterprise or fund-administration client.
  • The SMART IR pipeline spans four disconnected systems — this platform → Monday.com → a 20-sheet master workbook → Google Sheets/Slides — and takes 4–6 hours of manual work per company, per quarter.
Vision

The Unified Vision

The plan replaces two aging, disconnected systems (this platform + the manual scoring pipeline) with one platform serving every role — founder, staff, LP, GP, Champion, sponsor — through role-based views on a shared data model. SMART IR becomes a live, computed engine instead of a spreadsheet process, and Legacy Funds gets a full digital presence built on the same foundation, not a separate system.

What Gets Built

AngelSpan core rebuildFull feature parity plus a WYSIWYG editor, in-app publication preview, autosave, and per-stakeholder engagement analytics.

Milestones (not phases — see note above)

Milestone 1
Foundation & Parity

Platform modernization

Full feature parity on the new stack, identity and access control, and the shared data model — locked early enough for Legacy Funds work to start in parallel rather than waiting.

ℹ️

This milestone replaces the stack Phase 1 hardened, not builds on it. Phase 1's bug fixes, billing fix, and monitoring/alerting target the current Angular 1 / .NET Core 2.2 platform; once this milestone lands on React/.NET 8, that spend is largely redone rather than reused. Phase 1 still earns its keep in the meantime — it's what keeps the current platform trustworthy for however long it takes this milestone to get funded and built.

Milestone 2
SMART IR Automation

The scoring engine goes live

Automated composite scoring, tear sheet and quarterly report generation, and the interactive visualizations — replacing the 20-sheet master workbook.

Milestone 3
Legacy Funds

Runs in parallel, not after

Application portal, LP and GP dashboards, and fund administration — designed to run alongside Milestone 1 on the same shared foundation, not sequenced behind it.

Milestone 4
Polish & Launch

Unified launch

Security hardening, data migration from all legacy systems, and a single go-live for AngelSpan and Legacy Funds together.

Target: ~12 months to unified launch. Outer boundary: ~18 months at the same total budget, if scope or timing slips.

Strategy

AI Strategy

The source document is explicit that these are two separate things, on two separate timelines and risk profiles — worth keeping distinct here too.

AI as a platform feature — gated, not assumed

Any AI-facing feature must clear six tests before it goes on the roadmap: a named pain point, a measurable baseline, a defined target outcome, a mandatory human review step, defined fallback behavior if the AI is wrong or unavailable, and a data-security review of what gets shared with any third-party model.

The most concrete near-term example is AI-assisted Bell Mason scoring: the model reads a founder's monthly update, drafts a suggested score range with the specific quotes that support it, and a Performance Analyst reviews and finalizes rather than scoring from scratch. AngelSpan's SMART IR history across 100+ companies becomes the calibration set that validates the model against how PAs have actually applied the rubric in practice.

AI as a development accelerant — separate from the above

During the build itself, engineers would use tools like GitHub Copilot, Cursor, and Claude Code for boilerplate, test generation, and documentation. The one non-negotiable rule: authentication, access-control enforcement, the scoring engine's core logic, payment processing, and data migrations stay human-written and human-reviewed throughout — not delegated to AI assistance — given this is a platform handling real investor capital.

Status

Prototype Status

✅

This step is done. The plan called for a small set of throwaway prototypes, built before any production engineering, purely to validate requirements with stakeholders. The working mock already built — not client-ready, but functioning end-to-end — satisfies that requirement. No additional pre-engineering prototyping is planned.

Scope

What's Deferred

  • SOC 2 Type II certification — requires 12+ months of audit-ready operation; can't be shortcut regardless of budget.
  • Mobile application — web-first is the right priority for a B2B platform; the API-first architecture makes it an incremental build later, not a re-architecture.
  • AI-assisted scoring suggestions — a follow-on investment after Phase 2's rule-based automation is live and validated, not part of this build.
  • Public API / third-party integrations — the API-first architecture is being built, but external developer access is a post-launch policy decision.
  • Advanced LP analytics (IRR, MOIC, DPI) — needs fund administration live and generating real cash-flow data first.
  • CRM integration and internationalization — not current operational needs; revisited if the business grows into them.
