Phase 1

Harden + Stabilize

Make the platform we already have trustworthy — no new features

Overview

What This Raise Funds

✅

What this raise can realistically fund. Of the $500k expected in late Aug / early Sept, roughly $125k is set aside for technology once salaries, marketing, and debt are covered. The rough estimate to harden the platform is ~$100k — an estimate, not a fixed price, on a system this old. Moving on to Phase 2 or 3 would be funded by additional fundraising, not by this raise.

Overview

Timeline & Cost

Monthly spend rate — Aug '26–Jun '27
  • Money lands late Aug–early Sep 2026; spend ramps to ~$50k/mo for 2–3 months once the work is staffed up — a fixed, front-loaded effort, not an open-ended build
  • Microsoft retires services and API versions on its own schedule, not ours — TLS 1.0/1.1 support in Storage Accounts, .NET 9 support in App Service, and older Key Vault API versions are all being retired within the next 12 months. We don't believe any of these specifically affect us today, but the pattern is the point: aging infrastructure accumulates this kind of exposure continuously
  • We keep raising money in parallel, not after — the plan is to approach investors now, targeting more capital closing around the same November window
  • No new features ship: no Legacy Fund portal, no automated scoring reports, no fund dashboards — the team keeps handling those the same way it does today
  • Hardening is estimated to wrap up around November — roughly the same time our own math suggests the $500k is largely spent across salaries, marketing, debt, and tech combined
  • The runway is tighter than it looks: a Legacy Fund could realistically need the platform as early as Q1–Q2 2027 — as little as one to four months after this work finishes

Scope

What Gets Fixed

What This Actually Fixes — 3 Areas*

1 — Aging softwareThe two core systems the platform runs on have gone years without a security update from their makers. Bringing them current, plus fixing all 9 known bugs — including a documented billing fault that could block client submissions if it resurfaces.
2 — Email & billingToday, setting up a new client's email delivery is a manual, multi-day process that regularly derails onboarding. We also can't currently create new products or adjust pricing in our payment system. Both get fixed.
3 — Basic monitoringSimple alerting so we find out about a problem before a client does. The Extended tier adds real backup/recovery and a record of who changed what.
Decision

Benefits & Trade-offs

Benefits

  • Sized to the ~$125k set aside for tech; hardening is estimated around ~$100k. The other phases would need funding this raise doesn't provide
  • Aimed squarely at the biggest concern — the platform breaking while new clients are signing up
  • Doesn't gamble on landing an institutional partner — the work makes sense whether or not that materializes
  • Intended to buy time for the raise-and-build cycle without an avoidable outage getting in the way

Risks & Trade-offs

  • Adds no new capability — Legacy Fund applications, scoring reports, and fund dashboards all stay fully manual
  • Some of this spend won't carry forward if a full rebuild follows — the billing fix is the clearest example, but the monitoring and alerting added here don't survive it either: a full rebuild stands up its own logging, SSL automation, and audit trail for the new stack, so that spend gets redone, not reused
  • This ~$100k isn't extra money — it's competing for the same $500k as salaries, marketing, and debt payoff
  • Built for reliability at today's size (50–100 clients), not unlimited growth — a big surge in signups still needs Phase 2 or 3 money to handle properly
Note on the existing demo: the AI-generated prototype already built is a sales tool for showing investors what the platform could look like — it has no real backend and can't take on an actual paying client. It's neither this hardening work nor Phase 2's pilot; it's a separate, useful asset for fundraising conversations.
