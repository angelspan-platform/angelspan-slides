Decisions

How to Read a Decision Record

Every non-trivial product choice made during the prototype build is captured as a decision record in decisions/. Records are numbered sequentially from D-001 through D-038. Each record has four sections:

  • ContextWhat problem or question prompted the decision — why it needed to be made at all.
  • Options ConsideredWhat alternatives were evaluated, described plainly enough to understand the trade-offs.
  • ChoiceWhat was selected and, where relevant, implementation detail on how it works.
  • ConsequencesWhat follows from the choice: what it enables, what it constrains, what remains open.

Records also carry a metadata header with id, status, owner, last_updated, and dependencies. The dependencies field links to other decisions or design system components that this record builds on.

Reading references When you encounter a reference like "see D-003" in a specification, that reference points to the decision record that explains why the feature works the way it does — not just what it does.

Process

How the Prototype Was Built

The prototype was built using a structured human-in-the-loop process involving three components: Claude (AI assistant), Lovable (AI-powered prototype builder), and GitHub (version control). Understanding this process helps explain the structure of the specification files and why correction prompts exist.

The Workflow

1
Session setup
Each working session began by reading SESSION.md to re-establish current state — what had been built, what decisions were in effect, and what was next. This was the memory continuity mechanism across sessions.
2
Prompt drafting (Claude)
Claude drafted each Lovable prompt based on the master spec for the cluster being built, the relevant decision records, the design system, and the shared agent context. Claude played multiple roles during this drafting process — product manager, UX designer, frontend lead, backend lead, and technical architect — reviewing the prompt from each perspective before finalizing it, to catch conflicts and gaps before anything was sent to Lovable.
3
Human review
Adam reviewed each drafted prompt before it was sent. This included reading the prompt for correctness, asking clarifying questions, challenging assumptions, and approving or requesting changes. No prompt was sent to Lovable without human review and approval — the point where the human confirmed that the specification being sent matched the intended product behavior.
4
Delivery to Lovable
Approved prompts were copied into the Lovable.ai chat interface and submitted. Lovable interpreted the prompt and made code changes to the prototype, automatically committing them to the main branch on GitHub.
5
Verification
After Lovable completed the build, Adam reviewed the prototype at angelspan-bridge-flow.lovable.app to verify that the implementation matched what the prompt specified — a functional walkthrough checking that screens rendered correctly, flows worked as intended, and data behaved as expected.
6
Correction or next prompt
If verification revealed that Lovable had misinterpreted the prompt, partially executed it, or introduced a bug, a correction prompt was written (e.g., C6-lovable-prompt-A2-correction.md) and the cycle repeated from step 2. If verification passed, the session moved to the next prompt. SESSION.md was updated at the end of each session.
7
Decision capture
Whenever a non-trivial choice was made during the session — whether during prompt drafting, human review, or post-verification — a decision record was written and added to decisions/. This ensured that the rationale behind each product choice was captured at the moment it was made, not reconstructed later.

What This Means for the Handoff Package

The correction prompts in specifications/ are not a sign of failure — they are a record of the refinement process. Each correction reflects a place where the specification needed more precision or where the prototype tool's interpretation diverged from intent. Reading the A → B → correction sequence for a complex cluster (C8 is a good example, with five correction passes) gives a clear picture of where the hard product questions were and how they were resolved.

Most durable artifact The decision records in decisions/ represent the output of the human review step — the choices that were explicitly examined rather than left implicit. They capture intent, not implementation.

Each cluster has a master spec file (e.g., C4a-pa-flow.md) that defines the full intended behavior of that area — written for humans, not for the prototype tool. These are the authoritative product specifications.

Each cluster also has a set of Lovable prompt files (e.g., C4a-lovable-prompt-A.md, C4a-lovable-prompt-B.md) that were the actual instructions sent to the Lovable.ai prototype builder. These are implementation artifacts from the build process — useful for understanding how the prototype was constructed, but not product specifications. When a prompt file and a master spec conflict, the master spec is correct.

Correction prompts (files ending in -correction.md) record places where the first prompt produced an incomplete or wrong result. The pattern of corrections in a cluster is a useful signal — it shows where the specification was ambiguous or where the prototype tool made wrong assumptions that had to be overridden.

The Core Split

The Core Principle

The prototype was built to prove user experience and product decisions, not architecture. Every screen, flow, and interaction at angelspan-bridge-flow.lovable.app represents a validated product decision. The underlying technical implementation — authentication model, data stores, API surface, PDF generation, billing integration — was chosen for prototype convenience and is fully disposable.

Treat src/ as A high-fidelity wireframe, not as a starting point for production code.
The Core Split

What the Prototype Proves (Carry Forward)

These are durable product decisions, validated through the build and UAT round 1. They are documented in decisions/ and referenced here by ID.

User Model and Access Control

  • Ten roles across two brand contexts (AngelSpan and Legacy Funds). Full role definition and compatibility matrix in D-002.
  • Two branded login URLs, one auth backend — app.angelspan.com and app.thelegacyfunds.com, brand context determined by URL, context switcher in nav for multi-role users. See D-003.
  • Single platform, role-based views. One application; what you see is determined by your role. No separate portals requiring logout/login — a non-negotiable architecture principle.
  • Stakeholder phase model. Phase 1: magic link only. Phase 2: authenticated portal with LP conversion pathway. See D-004.
  • PA is the only internal staff role (no CC/CRA distinction). See D-032.

Information Architecture and UX Patterns

  • Dual-brand palette strategy. One design system, two color schemes. Spacing, typography, shadows, and border radii are identical across both brands. Colors differ and are swapped by applying a single CSS class — .brand-angelspan or .brand-legacy — to the root element. No component code changes between brand contexts. Full spec in D-001 and design-system/DESIGN_SYSTEM.md.
  • Slide-out panel pattern as the primary detail/editing surface across the platform. 680px width standard. See D-006.
  • Monthly Update screen consolidation. Single screen with phase-appropriate views rather than separate screens per workflow state. See D-006.
  • Artifact lifecycle model. Draft → Distributed (locked). Corrections via superseding document, not edits to distributed artifacts. See D-007.
  • PA read-only stakeholder access. Founder owns the distribution list. See D-008.
  • Shared company detail components. <CompanySmartIRDetail> and <CompanyPublicationsView> are shared parameterized components, not role-specific implementations. See D-035.

Service Tiers and Billing Model

  • Three service tiers: Essentials ($249/mo), Insights ($399/mo), Intelligence ($599/mo). SMART IR access is gated by tier.
  • Stripe Elements at signup, Stripe Customer Portal for Founder self-service. See D-012.
  • Intelligence plan is an obligation of Legacy Fund admission, not a benefit. Triggers at capital deployment. See D-031.

Legacy Funds Product Decisions

  • Legacy Fund founder eligibility cap is $800K (not $1M). §1244 protection rationale documented in specifications/LF-eligibility-canonical.md. See D-021.
  • LP management is GP-owned, not self-service. "Become an LP" CTA sends email to GP; it does not create a record. See D-022, D-025.
  • No capital calls. Legacy Funds is not a drawdown fund. Capital tracking uses three metrics only: Target Raise, Committed Capital, Deployed Capital. See D-037, D-038.
  • Champion workflow. Self-select from submitted application pool. Application locks to claiming Champion. Deny re-pools. Accept upgrades Founder plan to Intelligence. See D-019, D-030.
  • Fund Administrator Portal links. AngelSpan provides the link only — no SSO, no data ingestion. See D-036.
  • GP-to-LP quarterly reporting via platform is intentional. GP writes narrative only; LPs have direct dashboard access to portfolio data. See D-026.

SMART IR Visualizations

  • Bubble chart behavior. Full spec in D-033: GP Dashboard gets animated chart (temporal scrubber, trail lines, hover-to-pause); LP gets static chart with window selector only; PA Dashboard gets stage-filtered static chart.
  • GP Dashboard fund filter. Fund pills scope the bubble chart and Submission Health panel. Reference lines recalculate on filter change. See D-034.

Before Production

Decisions That Need Resolution Before Production

These are items flagged in the decision log or SESSION.md as prototype-acceptable but requiring resolution before the production build begins.

Open Decisions

  • D-038Target Raise editability. Whether a fund's Target Raise amount can be edited after creation is unresolved. Options and a recommendation are documented in decisions/D-038-capital-tracking-model.md. Must be decided before the production Fund Admin edit flow is implemented.
  • —Identity provider: Azure AD B2C vs. Auth0. Not yet finalized. The product decisions around authentication (D-003) are implementation-agnostic, but the development team will need this resolved early in Phase 0.
  • —Stripe: patch current integration vs. adopt Stripe Billing fully. The prototype implements the "full Stripe Billing" model (Elements + Customer Portal), but the legacy platform has known Stripe bugs. The dev team needs explicit direction before Phase 1 billing work begins.
  • —Monolith vs. modular services. Not yet finalized. The prototype's frontend-only architecture provides no signal on this.

Known Pre-Production Corrections

These were accepted as prototype limitations but must be corrected before production:

  • D-027LF Admin Champions screen. Currently retains Champion management UI. Correct to GP-owned (Champion management belongs to GP, LF Admin is read-only oversight). The prototype has this wrong.
  • D-035Founder /app/founder/smartir not yet using shared <CompanySmartIRDetail>. Deferred from C10. The production build should use the shared component from day one.
  • —LP fundsReceived toggle UX. Current implementation buries the toggle in an edit modal. This is a routine recurring action for GPs — it must be an inline row toggle accessible directly from the LP list.
  • —GP Fund Activity "Distributed" tile shows $0. Acceptable in prototype. Needs proper calculation in production.
  • D-008PA stakeholder access. Currently enforced only at the UI layer. The product decision is that PAs cannot modify Founder stakeholder lists — this must be enforced at the API level in production.
Before Production

How to Read the TypeScript Data Files

The prototype's in-memory data structures (src/lib/lf-data.ts, src/lib/admin-data.ts, etc.) are best understood as annotated product specifications in executable form. They tell you:

  • What entities exist and what fields they have
  • What relationships exist between entities (fund → companies → LPs → GPs)
  • What state machines exist (application status: Draft → Submitted → Under Review → Accepted/Denied)
  • What calculations are performed (slope, composite SMART IR, eligibility checks)

They do not tell you:

  • How those entities should be stored in a relational database
  • What indexes, constraints, or foreign keys apply
  • What should be computed at write time vs. read time
  • What belongs in the core SQL database vs. the analytics store
How to use them As a specification input to the data model design phase — not as a schema to migrate from.
